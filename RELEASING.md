# Releasing

This repository uses manual SemVer releases for HACS and Home Assistant, with `git-cliff` generating the GitHub Release notes.

## Version format

- Stable integration versions use `X.Y.Z` in `custom_components/oklahoma_gas_and_electric/manifest.json`.
- Beta integration versions use `X.Y.ZbN` in `custom_components/oklahoma_gas_and_electric/manifest.json`.
- Git tags and GitHub Release names use the same version with a leading `v`:
  - Stable: `vX.Y.Z`
  - Beta: `vX.Y.ZbN`

Examples:

- Stable:
  - Manifest version: `0.2.0`
  - Git tag: `v0.2.0`
  - GitHub Release: `v0.2.0`
- Beta:
  - Manifest version: `0.3.0b0`
  - Git tag: `v0.3.0b0`
  - GitHub Release: `v0.3.0b0`

## HACS and Home Assistant behavior

- HACS reads published GitHub Releases when releases exist. Tags alone are not enough.
- Stable releases must be normal GitHub Releases.
- Beta releases must be GitHub prereleases.
- HACS users only see beta updates if they enable prereleases for the repository.

`hacs.json` note:

- `homeassistant: "2026.6.0b0"` would mean this integration allows Home Assistant beta builds.
- It does not mark this integration itself as a beta release.
- Integration beta status comes from the integration version string plus the GitHub prerelease flag.

## Changelog policy

- `git-cliff` is the release changelog generator for this repository.
- The release workflow generates GitHub Release notes from git history and the current tag.
- This repository does not commit a `CHANGELOG.md` file right now.
- GitHub Release notes are the release changelog source of truth for now.
- If a committed `CHANGELOG.md` is added later, document that flow explicitly before using both.

## Release artifacts

- No release artifacts are required by default.
- This integration ships from the repository source layout in the GitHub Release.
- Do not add `zip_release` or packaged release artifacts unless the repository later chooses HACS zip-based releases.

## Maintainer workflow

1. Update `custom_components/oklahoma_gas_and_electric/manifest.json` to the next release version.
2. Commit the version change using `chore(release): vX.Y.Z`.
3. Push the commit to the release branch.
4. Create and push the release tag:
   - Stable: `git tag vX.Y.Z && git push origin vX.Y.Z`
   - Beta: `git tag vX.Y.ZbN && git push origin vX.Y.ZbN`
5. Let the GitHub Actions `Release` workflow create a draft GitHub Release with notes generated by `git-cliff`.
6. Open the draft GitHub Release and verify the generated changelog text before publishing it.
7. For beta releases, keep the GitHub Release marked as `pre-release`.
8. For stable releases, ensure the GitHub Release is not marked `pre-release`.

## CI workflows

- `.github/workflows/validate.yml` runs HACS validation.
- `.github/workflows/hassfest.yml` runs Home Assistant `hassfest`.
- `.github/workflows/release.yml` generates release notes with `git-cliff` and creates a draft GitHub Release for pushed `v*` tags or a manual workflow dispatch.
